# 第5章　設計フレームワーク

!!! abstract "この章のねらい"
    第Ⅰ部の理論を「**手順**」に変える。場当たり的に要素を足すのではなく、目的から逆算して設計するための代表的フレームワーク——Werbach の 6D、Chou の Octalysis——を実務で回せる形で解説する。最後に、設計ワークショップの進行テンプレートを示す。

## 5.1 Werbach の 6D（6ステップ・デザインプロセス）

Kevin Werbach が提示した、ゲーミフィケーション設計の標準的な手順[^werbach2012]。6つのステップの頭文字がすべて D で始まる。**目的（ビジネス成果）から始まり、要素は最後**という順序が肝心である。

### ステップ1: Define business objectives（ビジネス目標を定義する）

最初に、**何のためにやるのか**を明確にする。「エンゲージメント向上」では曖昧すぎる。具体的なビジネス成果（継続率、学習完了率、再訪頻度など）に落とす。

- 目標を列挙 → 優先順位づけ → **「手段」を目標に紛れ込ませない**（「バッジを導入する」は目標ではなく手段）。
- ここを飛ばすと、後で効果測定（第13章）ができなくなる。

### ステップ2: Delineate target behaviors（対象行動を明確にする）

目標を達成するために、ユーザーに**具体的に何をしてほしいか**を定義する。

- 行動は観測・測定可能な粒度に（例:「週3回ログインして10分学習する」）。
- 同時に**成功指標（メトリクス）**を定める。各行動が目標にどう貢献するかの仮説を立てる。

### ステップ3: Describe your players（プレイヤーを記述する）

ユーザーは誰か。第4章のタイプ論を使い、動機の多様性を把握する。

- 年齢・属性だけでなく、**動機の構成**（HEXAD等）を描く。
- ペルソナを作り、「この人は何で動くか」を言語化する。

### ステップ4: Devise activity loops（アクティビティ・ループを考案する）

行動を駆動する**ループ**を設計する（第3章）。

- **エンゲージメントループ** — 動機→行動→フィードバック→動機 の短期循環。
- **プログレッションループ** — 短期の積み重ねが長期の成長（レベル・アンロック）につながる構造。

### ステップ5: Don't forget the fun（楽しさを忘れない）

最も飛ばされやすく、最も重要なステップ。仕組みが「作業」になっていないかを問う。

- 自問:「報酬や義務を**すべて取り除いても**、この体験は楽しいか？」
- 楽しさの源泉（Marczewski/LeBlancの「楽しさの種類」——発見、達成、表現、社交、競争など）のどれを狙うかを意識する。

### ステップ6: Deploy the appropriate tools（適切なツールを実装する）

**ここでようやく**具体的なコンポーネント（ポイント・バッジ等、第3章/第6章）を選ぶ。1〜5で定義した目標・行動・プレイヤー・ループ・楽しさを実現する手段として選定する。

!!! tip "6Dの最大の教訓"
    多くの失敗は「ステップ6から始める」ことで起きる。「バッジを付けよう」ではなく、**目標→行動→人→ループ→楽しさ→要素**の順で考える。要素は**最後**。これは第3章「コンポーネントから考えるな」と同じ原則を、プロセスに落としたものだ。

## 5.2 Chou の Octalysis（8つのコアドライブ）

Yu-kai Chou が提唱した、人間の動機を**8つのコアドライブ**で捉えるフレームワーク[^chou2015]。設計の「**動機の抜け漏れ**」を発見するのに優れる。8角形（Octagon）の各頂点に1つずつドライブを配置する。

| # | コアドライブ | 内容 | 代表的な仕掛け |
|---|---|---|---|
| 1 | **叙事的意義・天命** | 自分は大きな何かの一部だという感覚 | 物語、大義、ビギナーズラック、選ばれた感 |
| 2 | **発展と達成** | 進歩し、克服し、上達する | ポイント、バッジ、レベル、進捗バー、クエスト |
| 3 | **創造性とフィードバック** | 創意工夫し、結果を見て試行錯誤する | カスタマイズ、ボーナス、自由な組み合わせ |
| 4 | **所有と保有** | 何かを所有・蓄積・改善する | コレクション、仮想財、アバター、ポイント残高 |
| 5 | **社会的影響と関係性** | 他者との比較・協力・つながり・羨望 | チーム、贈与、メンター、社会的証明 |
| 6 | **希少性と焦り** | 手に入りにくいから欲しくなる | 期間限定、アポイントメント、じらし |
| 7 | **予測不能性と好奇心** | 次に何が起こるか分からない | 変動報酬、ガチャ、ミステリーボックス、抽選 |
| 8 | **損失と回避** | 失いたくない、無駄にしたくない | ストリーク、期限切れ、サンクコスト |

### 2つの重要な軸

Octalysis の真価は、8ドライブを**2つの軸で分類**する点にある。

- **左脳ドライブ vs 右脳ドライブ**
    - **左脳（外発寄り）** — 発展と達成、所有と保有、希少性と焦り。論理・所有・分析に関わる。
    - **右脳（内発寄り）** — 創造性とフィードバック、社会的影響と関係性、予測不能性。感情・社交・好奇心に関わる。
    - → 左脳ドライブは外的報酬に近く、**報酬を止めると動機も止まりやすい**。右脳ドライブは活動自体が報酬になりやすく**持続的**。

- **ホワイトハット vs ブラックハット**
    - **ホワイトハット（上方）** — 叙事的意義、発展と達成、創造性。**前向きで力を与える**動機。長期的に健全。
    - **ブラックハット（下方）** — 希少性、予測不能性、損失と回避。**焦り・不安・強迫**で動かす。短期的に強力だが、使いすぎるとユーザーを消耗させ、離脱や不信を招く。

!!! warning "ブラックハットの扱い"
    ブラックハット・ドライブ（希少性・変動報酬・損失回避）は即効性が高く、つい多用したくなる。しかし**長期的にはユーザーを疲弊させ、倫理問題（第14章）にも直結**する。短期コンバージョンにブラックハットを使うなら、ホワイトハットで「**戻ってきたくなる理由**」を必ず併設する。健全な設計は両者のバランスで成り立つ。

### Octalysis の使い方

1. 既存または計画中の体験を、8ドライブそれぞれについて「どの仕掛けで満たしているか」を書き出す。
2. **欠けているドライブ**を見つける（多くの設計は「発展と達成」＝左脳に偏る）。
3. 右脳・ホワイトハットのドライブを補強できないか検討する。
4. ユーザー体験の**フェーズ別**（発見→オンボーディング→足場固め→終盤、第7章）に、各時点でどのドライブが効くべきかを設計する。

## 5.3 その他の主要な枠組み

目的に応じて使い分けられる、他の有用なフレームワークを簡潔に紹介する。

- **自己決定理論（SDT）を直接使う** — 第2章。最もシンプルで強力。「有能感・自律性・関係性を高めているか」という3問だけでも、多くの設計判断ができる。理論的厳密さを重視するならこれが土台。
- **MDA / 3層モデル** — 第3章。「体験から逆算」「コンポーネントは最後」を徹底する分析レンズ。
- **HEXAD** — 第4章。ユーザータイプから「誰に何が効くか」を設計する。
- **Lens of Intrinsic Skill Atoms（Deterding）** — 行動を「スキルの原子」に分解し、各スキルに対して目標・行動・フィードバック・課題が揃っているかを点検する、より精緻な内発的動機重視のアプローチ[^deterding2015]。
- **Player Journey（プレイヤージャーニー）** — 時間軸（オンボーディング→習熟→熟達）でユーザーの体験を設計する視点。第7章で詳述。

!!! tip "フレームワークの使い分け"
    フレームワークは互いに排他的ではなく、**レイヤーが違う**。実務では:

    - **プロセス（手順）が欲しい** → 6D
    - **動機の抜け漏れを発見したい** → Octalysis
    - **理論的な妥当性を担保したい** → SDT
    - **誰に効くかを設計したい** → HEXAD
    - **体験の質を点検したい** → MDA / Skill Atoms

    多くのプロジェクトは「**6Dで進め、各ステップでOctalysisとSDTで点検する**」という組み合わせがうまくいく。

## 5.4 設計ワークショップ進行テンプレート

チームでゲーミフィケーションを設計する際の、半日〜1日のワークショップ進行例。6D を骨格に、Octalysis/SDT を点検に使う。

!!! example "ワークショップ進行（約4〜6時間）"
    **準備（事前）**

    - 対象サービス・課題の共有、参加者（企画・デザイン・開発・データ）の招集。

    **セッション1: 目的と行動（6D ①②、60分）**

    1. ビジネス目標をポストイットで発散 → 投票で優先順位づけ。
    2. 各目標に対し「ユーザーにしてほしい**具体的行動**」を定義。
    3. 各行動の**成功指標**を仮置きする（第13章につなぐ）。
    4. ⚠️ チェック:目標に「手段（バッジ等）」が紛れていないか。

    **セッション2: ユーザー理解（6D ③、45分）**

    1. ペルソナを2〜3体作成。
    2. 各ペルソナの**動機構成**を HEXAD で記述。
    3. ⚠️ チェック:競争系だけで全員カバーできるか？（多動機設計）

    **セッション3: ループと楽しさ（6D ④⑤、60分）**

    1. エンゲージメントループとプログレッションループを図示。
    2. 「報酬を全部外しても楽しいか？」を全員で議論。
    3. 狙う「楽しさの種類」を1〜2個に絞る。

    **セッション4: 動機の点検（Octalysis、45分）**

    1. 現案を8コアドライブにマッピング。
    2. **欠けているドライブ**と**ブラックハット偏重**を発見。
    3. 右脳・ホワイトハットの補強案を出す。

    **セッション5: 要素の選定（6D ⑥、45分）**

    1. ①〜⑤を満たす**コンポーネント**を選ぶ（第6章のカタログ参照）。
    2. 各要素を SDT の3欲求で最終点検。
    3. MVP（最小構成）と将来拡張を分ける。

    **クロージング（30分）**

    - 効果測定計画（KPI・A/B、第13章）と、倫理レビュー（第14章）の宿題を確認。
    - プロトタイプとテストの段取りを決める。

!!! check "フレームワーク活用のチェック"
    - [ ] **目標→…→要素**の順で設計したか？（要素から始めていないか）
    - [ ] 成功指標を**設計と同時に**定義したか？
    - [ ] Octalysis で**欠けた動機**がないか点検したか？
    - [ ] **ブラックハット**に偏っていないか？ ホワイトハットの「戻る理由」はあるか？
    - [ ] 各要素を **SDTの3欲求**で説明できるか？
    - [ ] MVPと将来拡張を分け、**まず小さく試す**計画になっているか？

## この章のまとめ

- **Werbach 6D** は標準プロセス。目標→行動→プレイヤー→ループ→楽しさ→**要素（最後）**の順。
- **Chou Octalysis** は8コアドライブで**動機の抜け漏れ**を発見する。左脳/右脳・ホワイトハット/ブラックハットの軸が鍵。
- **ブラックハット**（希少性・変動報酬・損失回避）は即効だが消耗を招く。ホワイトハットで持続性を担保する。
- フレームワークはレイヤーが違うので**組み合わせる**。実務は「6Dで進め、Octalysis/SDTで点検」が有効。
- ワークショップ・テンプレートで、チーム設計を再現可能なプロセスにできる。

次章では、6Dの最終ステップで選ぶ「**ツール＝メカニクス**」を、カタログとして体系的に棚卸しする。
